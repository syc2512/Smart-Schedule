# 智能课程表 Pro — 后端系统架构设计规范

> 适用对象：当前 `smart-timetable-pro.html`（纯前端 + IndexedDB + 浏览器侧直连 AI）。
> 本文档可作为「系统提示词 / 架构规格书」直接交给实现团队或 AI 编码助手落地。
> 设计原则：安全优先、水平扩展、容错降级、可观测、最小权限。

---

## 0. 现状分析与核心痛点

| 现状 | 风险 | 后端要解决 |
|------|------|-----------|
| AI Key 存浏览器、前端直连 `/chat/completions` | Key 泄露、被刷量、无法统一限流 | 密钥服务端托管 + 统一代理 |
| 数据存 IndexedDB（本地） | 换设备丢失、无协作、无备份 | 中心化数据库 + 多端同步 |
| 单点 `fetch` + 简单 `AbortController` 超时 | 无重试/熔断，供应商抖动即失败 | 重试/超时/熔断/故障转移 |
| 无输入校验集中层 | XSS/注入依赖前端 `esc()`，不可信 | 服务端多层校验与清洗 |
| 无监控告警 | 故障靠用户反馈 | 指标/日志/链路/告警闭环 |

---

## 1. 架构设计目标

1. **高扩展性（水平扩展）**：无状态服务可随负载横向扩容；数据库读写分离；AI 任务异步化削峰。
2. **高稳定性**：目标可用性 ≥ 99.9%；任何第三方（AI 供应商）故障不影响核心课表功能。
3. **安全合规**：密钥零暴露、输入零信任、最小权限、传输与存储加密。
4. **可观测性**：指标 / 日志 / 链路三位一体，故障可定位、可预警。
5. **降级可用**：AI 全挂时，课程导入仍可走「模板/规则兜底」，核心读写不受影响。

---

## 2. 整体架构图（分层）

下方为分层架构图，自顶向下：客户端 → 边缘（WAF/网关/负载均衡）→ 无状态服务 → 基础设施（缓存/队列/对象存储/服务发现）→ AI 供应商故障转移链 → 数据层 → 可观测性。

![架构图](architecture-diagram.svg)

**图例说明**
- **边缘层**：WAF 抗 DDoS 与注入；API 网关（APISIX）统一鉴权、限流、路由、协议转换；L7 负载均衡把流量打到多副本服务。
- **服务层**：三个无状态服务（认证 / 课程表 / AI 代理），任意副本可水平扩容。
- **基础设施**：Redis（会话、限流计数、AI 语义缓存）、RabbitMQ（AI 异步任务与死信重试）、对象存储（上传的图片/PDF/XLSX）、服务发现（Provider 注册与健康探测）。
- **AI 故障转移链**（本系统重点）：AI 代理 → 熔断器 → 主供应商 → 备选供应商 → 本地模型 → 模板兜底。任一层失败自动降级。
- **数据层**：PgBouncer 连接池 → PostgreSQL 主库（写）+ 只读副本（读）。
- **可观测性**：Prometheus+Grafana（指标）、Loki/ELK（日志）、OpenTelemetry（链路）、AlertManager（告警）贯穿全部组件（虚线）。

## 3. 技术选型与理由

| 层 | 选型 | 理由 |
|----|------|------|
| 语言/框架 | **Node.js + TypeScript + NestJS** | 与前端同源便于共享 Zod 类型；异步 I/O 适合代理 AI 长连接；NestJS 内置 Pipe（校验）、Guard（鉴权）、Interceptor（日志），天然契合「输入校验 / 安全」要求。热点 AI 代理如需极致性能可换 Go。 |
| API 网关 | **Apache APISIX**（或 Kong） | 基于 etcd 动态配置，内置限流、JWT、WAF 插件、AI 限流插件；比 Nginx 更易做灰度与多租户。云上可用 ALB + 托管 API Gateway。 |
| 负载均衡 | **K8s Ingress(Nginx) + Service** | L7 路由 + 会话无关轮询；外部 L4 LB 做入口高可用。 |
| 缓存 | **Redis 7（哨兵/集群）** | 限流令牌桶、会话、AI 响应语义缓存（同输入命中缓存省成本）。 |
| 消息队列 | **RabbitMQ**（死信队列 + 重试） | AI 解析为耗时操作，入队异步处理；超时/失败进死信队列自动重试，避免请求线程被拖死。事件流审计可加 Kafka。 |
| 数据库 | **PostgreSQL 15+**（主 + 只读副本） | 强一致、JSONB 存课程弹性字段、`tsvector` 全文检索、`pgcrypto` 加密；读副本扛查询峰值；PgBouncer 池化连接。 |
| 服务发现 | **K8s DNS + Readiness 探针**（多云用 Consul） | 供应商/服务实例自动注册与健康探测，支撑故障转移。 |
| 熔断 | **opossum**（Node 熔断器） | 每供应商独立熔断器，滑动窗口统计失败率，自动开/半开。 |
| 校验 | **Zod + class-validator** | 运行时 schema 校验，字段白名单、类型/长度/边界约束，拒绝未知字段。 |
| 清洗/防 XSS | **DOMPurify / sanitize-html** | 富文本或回显内容做 HTML 净化；输出统一转义。 |
| 可观测 | **OpenTelemetry + Prometheus + Grafana + Loki + AlertManager + Sentry** | 指标/日志/链路统一，异常自动告警到钉钉/飞书/邮件。 |
| 部署 | **Docker + Kubernetes + HPA** | 镜像化、声明式扩缩容；HPA 按 CPU/请求数自动水平扩容。 |

## 4. 核心模块设计

### 4.1 API 网关层
- 统一入口，终止 TLS；强制 HTTPS（HSTS）。
- **鉴权**：校验 JWT，注入 `x-user-id` 头给下游（下游不再信任客户端身份）。
- **限流**：按用户令牌桶（如 60 req/min）+ 全局兜底；AI 接口单独更严格限额。
- **防护**：请求体大小上限（如 10MB）、WAF 规则（SQLi/XSS 特征）、非法路径 404。
- **路由**：`/api/v1/*` → 对应服务；`/ai/*` → AI 代理服务。

### 4.2 负载均衡
- 外部 L4 LB（云厂商/Keepalived）保证入口高可用。
- K8s Service 做服务间负载，多副本轮询；配合就绪/存活探针剔除坏节点。
- 无状态设计：任何副本可处理任意请求，扩容即生效。

### 4.3 缓存层（Redis）
- **会话/令牌**：短时效 JWT 黑名单、刷新令牌。
- **限流计数**：令牌桶计数器。
- **AI 语义缓存**：对 `(provider, model, 输入哈希)` 缓存解析结果，TTL 可配，显著降低供应商调用与成本。
- **热点课表**：读多写少，缓存 `timetable:{id}`，写时失效。

### 4.4 消息队列（RabbitMQ）
- 队列 `ai.import`：接收异步导入任务，消费者调用 AI 代理。
- 死信队列 `ai.import.dlq` + 指数退避重试（最多 3 次）。
- 任务状态表 `ai_jobs` 供前端轮询 / WebSocket 推送。

### 4.5 数据库设计（PostgreSQL）
见第 7 节 DDL。要点：UUID 主键、软删除、租户隔离（`owner_id`）、合理索引、连接池。

### 4.6 日志与监控
- **结构化日志**：JSON 格式，带 `trace_id`/`user_id`/`provider`，集中到 Loki/ELK。
- **指标**：请求量、P95 延迟、错误率、熔断状态、队列积压、缓存命中率。
- **链路**：OpenTelemetry 串联网关→服务→DB→AI 供应商。
- **告警**：错误率 >1%、熔断打开、队列积压、DB 连接耗尽 → AlertManager 即时通知。
- **错误追踪**：Sentry 捕获未预期异常。

## 5. 第三方 AI API 容错机制（重点）

设计目标：供应商抖动、限流、宕机时，**用户无感知或优雅降级**，且不计费浪费。

### 5.1 重试机制（Retry）
- 仅对**安全可重试**错误重试：`408/429/5xx/超时/连接重置`；`400/401/403` 不重试。
- 策略：**指数退避 + 抖动**（base 300ms，factor 2，max 3~5 次），避免重试风暴。
- 幂等：以 `请求哈希` 去重，防止重复计费。

```ts
async function withRetry<T>(fn: () => Promise<T>, opts: {
  retries?: number; base?: number; max?: number;
} = {}): Promise<T> {
  const retries = opts.retries ?? 4;
  let attempt = 0;
  while (true) {
    try { return await fn(); }
    catch (e) {
      const retryable = isRetryable(e);          // 429/5xx/timeout/ECONNRESET
      if (!retryable || attempt >= retries) throw e;
      const delay = Math.min(opts.max ?? 8000,
        (opts.base ?? 300) * 2 ** attempt + Math.random() * 200);
      await new Promise(r => setTimeout(r, delay));
      attempt++;
    }
  }
}
```

### 5.2 超时处理（Timeout）
- 每类调用独立超时：`text 30s / image 60s / file 90s`，用 `AbortController` + `Promise.race` 强制中断。
- 超时计入失败指标，并触发重试/熔断。

```ts
function withTimeout<T>(p: Promise<T>, ms: number, ctrl?: AbortController): Promise<T> {
  const ac = ctrl ?? new AbortController();
  const t = setTimeout(() => ac.abort(), ms);
  return Promise.race([
    p,
    new Promise<T>((_, rej) => ac.signal.addEventListener('abort',
      () => rej(new TimeoutError(`timeout ${ms}ms`))))
  ]).finally(() => clearTimeout(t));
}
```

### 5.3 熔断降级（Circuit Breaker）
- 每个供应商独立熔断器（opossum）：关闭 / 打开 / 半开三态。
- 阈值示例：滚动窗口 20 次调用，失败率 ≥ 50% → 打开；冷却 30s 后半开试探。
- 打开期间请求**直接短路**，不浪费连接，立即走故障转移。

```ts
const breaker = new CircuitBreaker(callProvider, {
  timeout: 30000, errorThresholdPercentage: 50,
  resetTimeout: 30000, rollingCountTimeout: 20000,
  rollingCountBuckets: 10,
});
breaker.on('open', () => metrics.inc('breaker_open', { provider }));
```

### 5.4 服务发现与故障转移（Failover）
- 供应商注册到「Provider Registry」，带健康分（最近成功率、延迟、配额余量）。
- 选择器按健康分排序；主供应商熔断或失败 → 自动切下一个健康供应商。
- 供应商类型：OpenAI 兼容 A、Azure/自建 B、本地模型 C；逐级兜底。
- 全失败 → 触发**模板/规则解析兜底**（正则解析常见课表文本），保证导入可用。

```ts
async function routeAI(req: AIRequest): Promise<Course[]> {
  const order = providerRegistry.healthy();        // 按健康分排序
  for (const p of order) {
    try { return await withTimeout(withRetry(() => breaker(p).fire(req)), p.timeout); }
    catch (e) { logger.warn(`provider ${p.name} failed`, { err: e.message }); }
  }
  return templateFallback.parse(req);               // 最后兜底，绝不抛给用户崩溃
}
```

### 5.5 错误日志与告警
- 每次失败记录结构化日志：`provider / 错误码 / 耗时 / trace_id / user_id`。
- 指标：各供应商 `error_rate`、`p95`、`breaker_state`、`quota_used`。
- 阈值告警：供应商错误率 >5%、熔断打开、配额 <10% → AlertManager 推送。
- 用户侧：返回友好错误 + 建议（如「AI 繁忙，已为你保存草稿，可稍后重试或手动添加」）。

### 5.6 限流与配额
- 网关级用户令牌桶；AI 代理维护**全局供应商配额**，避免触发供应商 429。
- 异步大任务入队，前端轮询/WebSocket 取结果，避免请求超时。

## 6. 数据安全与输入校验（多层防护）

原则：**零信任**——前端校验只是体验，服务端校验才是底线。

1. **边缘层**：WAF + 网关请求体大小上限 + 文件类型/大小白名单（图片/PDF/XLSX，≤10MB）。
2. **传输层**：全站 HTTPS，HSTS，安全 Cookie（`HttpOnly; Secure; SameSite`）。
3. **鉴权层**：JWT 校验 + RBAC；下游服务只信网关注入的 `x-user-id`，绝不信任客户端传入的 ID。
4. **应用层（校验）**：Zod DTO 校验，字段白名单、类型/长度/取值范围约束，拒绝未知字段；异常输入返回 400 并附字段级错误，**系统不崩溃**。
5. **清洗层**：所有回显文本统一转义；富文本用 DOMPurify 净化；剥离控制字符。
6. **持久层（防 SQL 注入）**：一律使用 ORM（Prisma/TypeORM）或参数化查询，**禁止字符串拼接 SQL**；数据库账号最小权限（仅 CRUD 所需表）。
7. **存储层**：上传文件存对象存储，返回签名 URL；MIME 与内容真实类型双重校验。
8. **输出层**：CSP、X-Content-Type-Options、X-Frame-Options 等响应头（等同前端已用 helmet 的思路）。
9. **租户隔离**：所有查询强制带 `owner_id` 过滤，防止越权访问他人课表。

**防 XSS 示例（回显）**
```ts
// 服务端净化（若存富文本）；普通文本直接转义后输出
import DOMPurify from 'isomorphic-dompurify';
const safe = DOMPurify.sanitize(userHtml, { ALLOWED_TAGS: ['b','i','br'], ALLOWED_ATTR: [] });
```

**防注入示例（参数化）**
```ts
// Prisma：自动参数化，杜绝注入
const courses = await prisma.course.findMany({
  where: { timetableId, name: { contains: q } },   // q 被安全绑定
});
```

## 7. 数据库 Schema（PostgreSQL）

```sql
-- 用户（密钥仅存派生哈希；AI Key 不再落前端）
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,            -- bcrypt/argon2
  display_name VARCHAR(100) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ NULL                     -- 软删除
);
CREATE INDEX idx_users_email ON users(email) WHERE deleted_at IS NULL;

-- 课表（租户隔离：owner_id）
CREATE TABLE timetables (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id UUID NOT NULL REFERENCES users(id),
  name VARCHAR(120) NOT NULL,
  semester VARCHAR(60),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ NULL
);
CREATE INDEX idx_tt_owner ON timetables(owner_id) WHERE deleted_at IS NULL;

-- 课程（弹性字段用 JSONB；时间/星期结构化便于冲突检测）
CREATE TABLE courses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  timetable_id UUID NOT NULL REFERENCES timetables(id),
  name VARCHAR(120) NOT NULL,
  teacher VARCHAR(80),
  location VARCHAR(120),
  day_of_week SMALLINT CHECK (day_of_week BETWEEN 1 AND 7),
  start_min SMALLINT CHECK (start_min >= 0 AND start_min < 1440),
  end_min   SMALLINT CHECK (end_min   >  start_min),
  color VARCHAR(20),
  credit NUMERIC(3,1),
  note TEXT,
  meta JSONB DEFAULT '{}'::jsonb,                 -- 扩展字段
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ NULL
);
CREATE INDEX idx_courses_tt ON courses(timetable_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_courses_time ON courses(day_of_week, start_min) WHERE deleted_at IS NULL;

-- AI 任务（异步导入状态）
CREATE TABLE ai_jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  status VARCHAR(20) NOT NULL DEFAULT 'pending',  -- pending/running/done/failed
  input_type VARCHAR(20),
  provider VARCHAR(40),
  result JSONB,
  error TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  finished_at TIMESTAMPTZ
);
CREATE INDEX idx_aijobs_user ON ai_jobs(user_id, created_at DESC);

-- AI 语义缓存（同输入命中，省成本）
CREATE TABLE ai_cache (
  key VARCHAR(64) PRIMARY KEY,                    -- sha256(provider|model|input)
  response JSONB NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 8. 关键模块接口定义

### 8.1 认证
```
POST /api/v1/auth/register   { email, password, displayName }
POST /api/v1/auth/login      { email, password } -> { accessToken, refreshToken }
POST /api/v1/auth/refresh    { refreshToken }    -> { accessToken }
```
- 密码强度校验（长度≥8、含大小写/数字）；`password_hash` 用 argon2/bcrypt。

### 8.2 课表与课程（租户隔离，所有写操作校验 owner）
```
GET    /api/v1/timetables
POST   /api/v1/timetables            { name, semester }
GET    /api/v1/timetables/:id/courses
POST   /api/v1/timetables/:id/courses { name, teacher, location, dayOfWeek,
                                        startMin, endMin, color?, credit?, note? }
PUT    /api/v1/courses/:id
DELETE /api/v1/courses/:id
```
- 服务端二次校验 `dayOfWeek∈[1,7]`、`endMin>startMin`，并做**冲突检测**返回 `conflicts`。

### 8.3 AI 导入（核心容错路径）
```
# 同步（小文本）
POST /api/v1/ai/import
  body: { type: 'text'|'image'|'pdf'|'xlsx', content: <string|base64>, fileName? }
  -> 200 { courses: Course[] }            // 经 重试/超时/熔断/故障转移/兜底
  -> 4xx { code, field? }                 // 输入校验失败
  -> 502 { code: 'AI_UNAVAILABLE', hint } // 全链路失败但已兜底/友好提示

# 异步（大文件/长任务）
POST /api/v1/ai/import/async  -> 202 { jobId }
GET  /api/v1/ai/jobs/:jobId   -> { status, result?, error? }
```
- 内部调用第 5 节的 `routeAI`；命中语义缓存直接返回。

### 8.4 AI 供应商内部接口（便于扩展）
```ts
interface AIServiceProvider {
  name: string;
  health(): Promise<{ ok: boolean; latencyMs: number }>;
  parse(req: AIRequest, signal: AbortSignal): Promise<Course[]>;
}
// 实现：OpenAICompatibleProvider / AzureProvider / LocalModelProvider
// 注册：providerRegistry.register(provider)
```

## 9. 部署与水平扩展

- **容器化**：每个服务独立 Docker 镜像；K8s Deployment 多副本。
- **自动扩缩**：HPA 按 CPU/请求 QPS 扩容；AI 消费者可独立扩缩（消费队列）。
- **多可用区**：主库 + 跨 AZ 只读副本；Redis 哨兵；RabbitMQ 镜像队列。
- **配置与密钥**：配置中心（etcd/Consul），AI Key 存密钥管理（Vault / 云 KMS），**永不进代码或前端**。
- **蓝绿/金丝雀**：网关按权重路由，平滑发布。

## 10. 容灾与备份

- PostgreSQL：WAL 持续归档 + 每日全量；RPO < 5min，RTO < 15min。
- 对象存储：跨区复制；Redis 可做 AOF 持久化（缓存可丢，影响性能非数据）。
- 演练：定期混沌测试（随机杀 AI 供应商/副本）验证熔断与降级。

## 11. 落地路线图（建议）

1. **MVP**：网关 + Auth + Timetable CRUD + PostgreSQL + 基础校验/限流。
2. **AI 代理**：Provider 抽象 + 重试/超时/熔断/故障转移 + 语义缓存 + 异步队列。
3. **加固**：WAF、CSP、租户隔离审计、Sentry、结构化日志。
