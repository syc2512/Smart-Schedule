# 智能课程表 Pro — 后端系统架构提示词（可直接复制给实现 AI）

> 使用方式：将下方代码块中的全部内容作为 **System Prompt** 交给 AI 编码助手（或作为团队架构任务书），即可按本架构落地后端。所有设计决策均来自 `backend-architecture-spec.md`，此处浓缩为可执行的单一提示词。

```
你是一位资深后端架构师。请为「智能课程表 Pro」（当前为纯前端 HTML 应用 smart-timetable-pro.html，数据存浏览器 IndexedDB，AI Key 与 AI 调用都在前端）设计并实现一套完整、可靠、可水平扩展的后端系统。

【背景与痛点】
- AI Key 存浏览器、前端直连 /chat/completions：密钥泄露、被刷量、无法统一限流 → 改为密钥服务端托管 + 统一代理。
- 数据存本地：换设备丢失、无协作 → 改为中心化 PostgreSQL + 多端同步。
- 单点 fetch + 简单超时：无重试/熔断 → 改为重试/超时/熔断/故障转移。
- 无服务端校验：XSS/注入依赖前端 esc() → 改为服务端零信任多层校验。
- 无监控：故障靠用户反馈 → 改为指标/日志/链路/告警闭环。

【硬性要求（必须逐条满足）】
1) 架构目标
   - 高扩展：无状态服务 + K8s HPA 水平扩容；PostgreSQL 读写分离；AI 任务异步化削峰。
   - 高稳定：可用性 ≥ 99.9%；任何第三方（AI 供应商）故障不影响核心课表读写。
2) 第三方 AI API 容错（重点）
   - 重试：仅对 408/429/5xx/超时/连接重置重试；指数退避 + 抖动；以请求哈希幂等去重，防重复计费。
   - 超时：text 30s / image 60s / file 90s，AbortController + Promise.race 强制中断；超时计入失败指标。
   - 熔断：每个供应商独立熔断器（opossum），滑动窗口统计失败率，自动开/半开/关闭。
   - 服务发现与故障转移：Provider Registry 带健康分（最近成功率/延迟/配额余量），主供应商熔断或失败自动切下一健康供应商，直至模板/规则兜底。
   - 日志与告警：结构化日志含 trace_id/user_id/provider；供应商错误率>5%、熔断打开、配额<10% 触发 AlertManager 告警。
3) 数据安全与输入校验（零信任多层防护）
   - 边缘：WAF + 网关请求体大小上限 + 文件类型/大小白名单（图片/PDF/XLSX，≤10MB）。
   - 传输：全站 HTTPS + HSTS；安全 Cookie（HttpOnly; Secure; SameSite）。
   - 鉴权：JWT + RBAC；下游服务只信网关注入的 x-user-id，绝不信任客户端传入 ID。
   - 应用层：Zod DTO 字段白名单校验，类型/长度/取值范围约束，拒绝未知字段；异常输入返回 400 并附字段级错误，系统不崩溃。
   - 清洗/输出：所有回显转义；富文本用 DOMPurify 净化；ORM 参数化防 SQL 注入；CSP / X-Content-Type-Options / X-Frame-Options 响应头。
   - 租户隔离：所有查询强制带 owner_id 过滤，防越权。
4) 核心架构组件
   - APISIX 网关（鉴权/限流/路由/协议转换）
   - L7 负载均衡（K8s Ingress + Service，多副本轮询 + 探针剔除坏节点）
   - Redis 7（会话/限流令牌桶/AI 语义缓存/热点课表缓存）
   - RabbitMQ（异步 AI 任务队列 + 死信队列指数退避重试）
   - PostgreSQL 15+ 主从（PgBouncer 连接池，JSONB 存课程弹性字段）
   - 可观测：Prometheus + Grafana + Loki + OpenTelemetry + AlertManager + Sentry
5) 交付物（必须全部给出）
   - 架构图：分层拓扑图 + AI 故障转移时序图（见 architecture-diagram.svg / ai-failover-sequence.svg）
   - 技术选型理由表（逐项说明为什么选该组件）
   - PostgreSQL DDL（users / timetables / courses / ai_jobs / ai_cache，含索引与约束）
   - 各模块接口定义：认证、课表 CRUD、AI 导入（同步 + 异步轮询）、AI Provider 内部接口
   - K8s HPA 与部署/容灾说明

【技术选型】
Node.js + TypeScript + NestJS；Apache APISIX；Redis 7；RabbitMQ；PostgreSQL 15+；opossum（熔断）；Zod（校验）；Prisma（ORM，参数化防注入）；Docker + Kubernetes + HPA。

【输出要求】
给出可运行的骨架代码（含重试/超时/熔断/故障转移的关键实现）与完整配置文件，并清晰说明水平扩展策略与容灾备份（RPO/RTO）方案。
```
