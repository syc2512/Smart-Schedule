# 智能课程表 Pro — 后端架构建议报告

> 分析对象：`smart-timetable-pro.html`（单文件前端，原生 JS + IndexedDB + PWA）
> 角色：后端架构师
> 日期：2026-07-17

---

## 1. 前端代码分析：功能需求与数据交互方式

### 1.1 核心数据契约（Course 实体）

前端通过 `normalize()` / `stripAll()` 固化了课程对象的字段，且 `test_data.js` 的 `mockCourses` 注释明确对齐「未来 `GET /api/courses` 的返回值」：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 课程唯一键（前端用 `c+时间戳+随机`） |
| `name` | string | 课程名（必填） |
| `teacher` | string | 教师 |
| `location` | string | 上课地点 |
| `weekday` | int 1–7 | 星期几 |
| `startPeriod` / `endPeriod` | int 1–12 | 起止节次 |
| `weekType` | enum | `all` / `odd` / `even`（每周/单周/双周） |
| `colorIndex` | int | 颜色索引（展示用） |

**结论**：后端只要返回/接收这个 9 字段结构，前端渲染逻辑**零改动**即可切换为真实数据。

### 1.2 当前的数据交互方式（全本地，无自有后端）

- **存储层**：`StorageAdapter` 用 IndexedDB 为主、localStorage 兜底（`loadCourses` / `saveCourses`）。所有 CRUD 都在浏览器内完成。
- **表单提交**：新增/编辑课程（`openForm` → `saveBtn`）只构建一个本地对象并调用 `save()`，**没有任何 fetch/XHR 发往自有服务器**。
- **启动加载**：`boot()` 读本地存储 → 无数据则用 `DATA.seedCourses` 种子数据；`test_data.js` 的 `mockCourses` 也是本地硬编码。

> 也就是说，这套前端的**核心功能目前完全不需要后端**。需要后端是为了「跨设备同步 / 云端备份 / 共享课表 / 集中式 AI」。

### 1.3 AI 调用方式（浏览器直连 LLM）

`AIAdapter` 是唯一的网络出口，且走**用户浏览器直连 OpenAI 兼容接口**：

```
POST {aiBase}/chat/completions
  Authorization: Bearer {aiKey}   // 用户本机填写，直连第三方
  body: { model, messages, temperature:0 }
```

能力：`parseText`（文本）、`parseImage`（图片）、`parseFile`（PDF/Excel→SheetJS 转 CSV 后发文本）。
**安全隐患**：Key 存在 localStorage，浏览器直接把 Key 发给第三方 → 无法集中限流/成本控制/审计，Key 泄露风险完全由用户承担。

### 1.4 渲染 / 导出 / 其他交互

- **渲染**：周视图/日视图纯前端计算（节次定位、冲突检测 `computeConflicts`、空闲块 `idleBlocks` 全在客户端）。
- **导出**：客户端生成 RFC5545 `.ics`（单双周拆独立重复事件）、JSON 备份下载、JSON 恢复上传 —— 全本地。
- **技术栈**：原生 JS（无框架、无构建步骤）、IndexedDB、SheetJS（CDN）、PWA（`manifest.webmanifest`）、移动优先。

## 2. 后端架构方案（3 选）

前端是「本地优先」范式，所以后端的目标不是替代本地存储，而是**补齐本地做不到的事**：跨设备同步、云端备份、共享课表、集中式 AI 网关、用户体系。下面三个方案按「投入/能力」递增排列。

### 方案 A — 本地优先 + 轻量 AI 代理（最小改动，零账号）

```
[ 浏览器 PWA ] ──► [ AI Proxy 微服务 ] ──► [ OpenAI 兼容 LLM ]
   (IndexedDB)        (持有 Key / 限流 / 审计)
```

- **形态**：保留现有 IndexedDB 全本地存储；只新增一个**极小的 AI 代理服务**（如 Cloudflare Worker / 单文件 Node），后端持有学校/平台的 LLM Key，做速率限制与成本封顶，前端不再存 Key。
- **适用场景**：只想解决「用户没有自己的 AI Key 也能用导入」；产品仍是单设备、个人使用、无账号。
- **优点**：改造成本最低（前端只改 `AIAdapter` 的 base url）；隐私好（课程数据不出本机）；运维几乎为零。
- **缺点**：**无跨设备同步、无云端备份、无共享课表、无用户体系**；同步类需求无法覆盖。

### 方案 B — 容器化 REST API（Node + PostgreSQL + Redis）

```
[ PWA ] ──► [ API Gateway / LB ] ──► [ AuthSvc + CourseSvc ]
                                        │          │
                                   [ PostgreSQL ] [ Redis 缓存/限流 ]
                                   [ Object Storage: ics/备份 ]
```

- **形态**：常规单体/微服务 REST 后端。`GET/POST/PUT/DELETE /api/courses` 直接映射前端数据契约；加 JWT 登录、云端备份、可选 AI 代理。`PostgreSQL` 存结构化课程，`Redis` 做缓存与限流。
- **适用场景**：需要**多设备同步 + 用户账号 + 云端备份**，且团队倾向自托管 / 完全可控；未来可能做校内共享课表。
- **优点**：技术栈最主流、人才好招、调试直观；PostgreSQL 的约束（`CHECK(weekday between 1 and 7)`）能在数据库层兜底数据质量；易扩展为微服务。
- **缺点**：需要运维（容器编排/数据库备份）；冷启动与扩缩容不如 Serverless 自动；初期成本高于方案 A。

### 方案 C — Serverless BFF + AI 网关（推荐）

```
[ PWA ] ──► [ Edge BFF (Workers/Vercel) ] ──┬─► [ PostgreSQL/Neon 或 D1 ]
   IndexedDB 本地缓存                         ├─► [ AI Gateway: 持有Key/限流/成本帽 ]
   (离线可用, 后台增量同步)                    └─► [ KV/Redis: 会话/限流令牌 ]
```

- **形态**：边缘 BFF（Cloudflare Workers / Vercel Functions）做「为前端量身定制的后端」，聚合并封装底层能力；数据落 Postgres（Neon 等托管）或 D1；AI 走同一网关集中代理。前端保持本地优先（IndexedDB 作为离线缓存 + 乐观更新），后台做**增量同步**。
- **适用场景**：希望**低运维、自动扩缩、全球低延迟**，同时又要同步、账号、共享、集中 AI；产品面向多校区/大规模学生。
- **优点**：零服务器运维、按需计费、边缘低延迟；本地优先保证弱网/离线可用，体验不降级；AI 网关统一管控 Key 与成本；与前端 PWA 理念天然契合。
- **缺点**：边缘运行时有冷启动与执行时长限制（需把重计算留给 DB/AI）；调试与本地复现略复杂；厂商绑定需通过 IaC 缓解。

## 3. 三方案对比

| 维度 | A. 本地优先+AI代理 | B. 容器化 REST | C. Serverless BFF + AI网关 |
|------|------|------|------|
| 跨设备同步 | ✗ | ✓ | ✓ |
| 用户/账号体系 | ✗ | ✓ | ✓ |
| 云端备份 | ✗（仍 JSON 文件） | ✓ | ✓ |
| 集中式 AI（无自有 Key） | ✓ | ✓ | ✓ |
| 共享/全校课表 | ✗ | ✓（易扩展） | ✓（易扩展） |
| 运维成本 | 极低 | 中（需管容器/DB） | 极低（托管） |
| 改造前端工作量 | 极小 | 中 | 中（加同步层） |
| 离线/弱网体验 | 最好（全本地） | 依赖网络 | 最好（本地优先+后台同步） |
| 成本模型 | 几乎为零 | 常驻开销 | 按量付费 |
| 推荐指数 | 做 MVP/演示 | 自托管团队 | 规模化产品 ⭐ |

## 4. 推荐方案（C）落地细节

### 4.1 数据模型（PostgreSQL）

```sql
CREATE TABLE courses (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name          VARCHAR(120) NOT NULL,
    teacher       VARCHAR(80)  NOT NULL DEFAULT '',
    location      VARCHAR(120) NOT NULL DEFAULT '',
    weekday       SMALLINT     NOT NULL CHECK (weekday BETWEEN 1 AND 7),
    start_period  SMALLINT     NOT NULL CHECK (start_period BETWEEN 1 AND 12),
    end_period    SMALLINT     NOT NULL CHECK (end_period BETWEEN start_period AND 12),
    week_type     VARCHAR(8)   NOT NULL CHECK (week_type IN ('all','odd','even')) DEFAULT 'all',
    color_index   SMALLINT     NOT NULL DEFAULT 0,
    version       BIGINT       NOT NULL DEFAULT 1,            -- 乐观锁/同步用
    updated_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at    TIMESTAMPTZ  NULL                           -- 软删，配合前端撤销
);
CREATE INDEX idx_courses_owner ON courses(owner_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_courses_owner_upd ON courses(owner_id, updated_at);
```

> `version` + `updated_at` 支撑**增量同步**（前端只拉 `updated_at > 上次同步点` 的变更），数据库层 `CHECK` 约束把前端 `clamp/normalize` 的兜底逻辑前移到服务端，双向保护数据质量。

### 4.2 API 契约（与前端零改动对齐）

```
GET    /api/courses?since=<ts>     # 返回 { courses:[...], serverTime }，since 用于增量同步
POST   /api/courses                # body 即前端 Course 9 字段（不含 id）
PUT    /api/courses/:id            # 带 version 做乐观并发控制
DELETE /api/courses/:id            # 软删（deleted_at）
POST   /api/courses/sync           # 批量上报本地变更，返回冲突解决后的权威结果
POST   /api/ai/parse               # 文本/图片/文件 → 课程数组（走服务端 AI 网关，前端不再持 Key）
GET    /api/me/export?fmt=ics      # 服务端生成 .ics（可选，前端当前已本地生成）
POST   /auth/login  /auth/refresh  # JWT
```

### 4.3 同步策略（本地优先的关键）

- 前端 `IndexedDB` 作为**离线真相源 + 乐观更新**；任何写操作先落本地、立即渲染，再后台 POST。
- 采用 **「最后写入获胜 + version 冲突检测」**：上传时带 `version`，服务端若 `version` 不匹配则返回 409 与当前权威值，前端弹「保留我的/采用云端」选择（或按 `updated_at` 自动合并）。
- 拉取用 `GET /api/courses?since=`，只同步增量，省流量、适合手机弱网。

### 4.4 AI 网关安全（替换浏览器直连）

- 前端不再存 Key，改为调用 `POST /api/ai/parse`；服务端持有 Key 于密钥库（如 Cloudflare Secrets / Vault）。
- 服务端做：**速率限制（每用户/每 IP）、Token 成本帽、请求体大小限制、对 LLM 返回做 JSON Schema 校验**后再回传（防注入/脏数据）。
- 超时与降级逻辑保留前端现有 `NO_KEY`/失败兜底文案，体验连续。

## 5. 与前端技术栈的配合

- **原生 JS / 无框架**：BFF 返回纯 JSON，前端 `fetch` 直接消费，无需 GraphQL 客户端等额外依赖；若用方案 B，REST 同样零额外负担。
- **IndexedDB 本地优先**：方案 C 的「本地缓存 + 后台同步」与现有 `StorageAdapter` 几乎同构，只需把 `saveCourses` 的「写入后」钩子加上一个后台同步调用，改动集中在 `StorageAdapter`，不侵入渲染逻辑。
- **PWA / 移动优先**：Edge BFF 天然支持 HTTPS、低延迟、离线缓存（可加 Service Worker 缓存 API 响应），与 PWA 理念一致。
- **SheetJS（已 CDN 引入）**：Excel 解析仍放前端（省带宽），只把转出的文本发给 `/api/ai/parse`；PDF/图片由前端读为 base64 上传，服务端转发 AI 网关。
- **AI 切换成本**：前端只需把 `AIAdapter.call` 的 `base` 从用户填写的 `aiBase` 改为平台网关 `/api/ai`，并去掉 `Authorization: Bearer {aiKey}` 头（改由服务端注入）—— 改动 < 10 行。

## 6. 落地建议（下一步）

1. **先确认产品目标**：仅个人工具 → 方案 A；要账号/同步/共享 → 方案 C（推荐）。
2. **数据契约锁定**：以本报告 §1.1 的 9 字段 + §4.1 的 `version/updated_at` 为准，前后端共用一份 schema 文档。
3. **最小可跑切片**：先实现 `GET/POST/PUT/DELETE /api/courses` + JWT，让前端 `StorageAdapter` 接上真实接口（按 `test_data.js` 注释路径），跑通「添加课程→刷新仍在」。
4. **再加 AI 网关**：把浏览器直连改为服务端代理，统一限流与成本控制。
5. **最后做同步层**：增量 `sync` + 冲突解决，前端 `IndexedDB` 升级为离线缓存。

> 一句话：**前端已为后端预留好接口（数据契约稳定、存储层可插拔），最务实的路径是方案 C（Serverless BFF + AI 网关），既保留本地优先的离线体验，又补齐同步/账号/集中 AI，且运维成本最低。**
