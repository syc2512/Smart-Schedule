# 统一日志系统

> 为智能课程表 Pro 后端服务提供完整的日志能力：控制台彩色实时输出 + 文件按时间/大小滚动持久化 + 容错写入 + 配置化级别/路径。

---

## 一、设计目标

| 需求 | 实现 |
|------|------|
| 控制台实时输出 | `StreamHandler` + ANSI 彩色 Formatter，便于开发调试 |
| 文件持久化 | `TimedRotatingFileHandler`（按天滚动）/ `RotatingFileHandler`（按大小滚动） |
| 统一格式 | `时间戳 \| 级别 \| 模块 \| [request_id] \| 消息` |
| 滚动切分 | 按天滚动 + 保留 N 份 + 旧文件 gzip 压缩；按大小 10MB 滚动可选 |
| 容错处理 | `SafeHandlerWrapper` 包装所有 handler，写入失败降级到 stderr，绝不影响主业务 |
| 可配置 | `logging_config.json` + 环境变量，支持 dev/test/prod 三环境预设 |

---

## 三、日志格式

### 控制台（彩色，dev 默认开启）

```
2026-07-18 13:25:14 | INFO    | app              | [req-abc-123] | POST /api/auth/register
```

字段说明：
- `时间戳`：精确到毫秒，本地时区
- `级别`：7 位左对齐（DEBUG/INFO/WARNING/ERROR/CRITICAL）
- `模块`：16 位左对齐（app/db/auth/crypto/llm_client 等）
- `[request_id]`：12 位 hex，每请求生成；无上下文时显示 `-`
- `消息`：业务消息，支持 `%s` 格式化 + `|` 分隔的结构化字段

### 文件（无颜色，纯文本，prod 默认）

文件路径：`logs/app.log`（主日志） + `logs/error.log`（仅 ERROR，prod 启用）

格式与控制台一致，但去除 ANSI 颜色码，便于日志采集工具（Filebeat/Fluentd）解析。

## 四、配置

### 4.1 配置文件 `logging_config.json`

```json
{
  "format": "%(asctime)s | %(levelname)-7s | %(name)-16s | [%(request_id)s] | %(message)s",
  "console_enabled": true,
  "console_color": true,
  "file_enabled": true,
  "file_rolling": "time",          // time | size | both
  "file_when": "midnight",          // TimedRotating: S/M/H/D/midnight/W0-W6
  "file_interval": 1,
  "file_backup_count": 30,
  "file_max_bytes": "10MB",         // 支持 "10MB" / "1GB" 字符串
  "error_file_enabled": true,
  "gzip_old_logs": true,

  "dev":  { "level": "DEBUG", "file_backup_count": 7,  "error_file_enabled": false },
  "test": { "level": "INFO",  "file_backup_count": 14 },
  "prod": { "level": "INFO",  "file_backup_count": 30, "console_color": false }
}
```

### 4.2 环境变量（优先级最高，覆盖配置文件）

| 变量 | 默认 | 说明 |
|------|------|------|
| `APP_ENV` | `dev` | 环境预设：dev/test/prod |
| `LOG_LEVEL` | 由 env 决定 | DEBUG/INFO/WARN/ERROR |
| `LOG_DIR` | `<项目根>/logs` | 日志文件目录 |
| `LOG_CONSOLE` | `true` | 是否启用控制台输出 |
| `LOG_FILE` | `true` | 是否启用文件输出 |

### 4.3 环境预设

| 环境 | 级别 | 控制台彩色 | 文件保留 | ERROR 单独文件 |
|------|------|------------|----------|----------------|
| `dev` | DEBUG | 是 | 7 天 | 否 |
| `test` | INFO | 是 | 14 天 | 是 |
| `prod` | INFO | 否 | 30 天 | 是 |

## 五、滚动策略

### 5.1 按时间滚动（默认，`file_rolling=time`）

使用 `TimedRotatingFileHandler`：
- `when=midnight`：每天 0 点滚动
- 滚动后旧文件重命名为 `app.log.YYYY-MM-DD`
- 启用 `gzip_old_logs=true` 时，自动 gzip 压缩为 `app.log.YYYY-MM-DD.gz`（约节省 80% 空间）
- 保留 `file_backup_count` 份，超出自动删除

### 5.2 按大小滚动（`file_rolling=size`）

使用 `RotatingFileHandler`：
- `max_bytes=10MB`（也支持 "1GB" 字符串）
- 滚动后命名为 `app.log.1`、`app.log.2` ...
- 保留 `backup_count` 份
- 适合高频日志场景

## 六、容错设计

### 6.1 `SafeHandlerWrapper`

所有 handler 都用 `SafeHandlerWrapper` 包装，提供三层防护：

```python
class SafeHandlerWrapper(logging.Handler):
    MAX_ERRORS_BEFORE_DISABLE = 10

    def emit(self, record):
        if self._disabled:
            return  # 1) 累计错误超阈值，自动禁用该 handler
        try:
            self._wrapped.emit(record)
        except Exception as e:
            self._on_emit_error(e, record)  # 2) 捕获异常，不向上抛
```

容错行为：
1. **首次失败**：打印一次醒目告警到 stderr（不递归到 logging）
2. **每次失败**：把当前日志降级输出到 stderr（前缀 `FALLBACK |`）
3. **累计 10 次失败**：自动禁用该 handler，避免后续重复报错

### 6.2 文件目录创建失败

```python
try:
    os.makedirs(log_dir, exist_ok=True)
except Exception as e:
    sys.stderr.write("[logger] 警告：创建日志目录失败 %s: %s\n" % (log_dir, e))
    return  # 跳过文件 handler，仅保留控制台 handler
```

### 6.3 测试验证

```
LOG_DIR="Z:/nonexistent/path" python app.py
```

输出：
```
[logger] 警告：创建日志目录失败 Z:\nonexistent\path: [WinError 3] 系统找不到指定的路径。
2026-07-18 13:28:42 | INFO | root | [-] | 日志系统初始化完成 | env=prod level=INFO dir=Z:\nonexistent\path
2026-07-18 13:28:42 | INFO | fault_test | [-] | 开始容错测试
=== 主流程未崩溃 ===
```

主业务流程完全不受影响。

## 七、上下文绑定（request_id）

### 7.1 用法

```python
from logger import get_logger, log_context

log = get_logger("app")

# 为请求绑定上下文（线程内有效）
with log_context(request_id="abc123", user_id=42):
    log.info("处理请求")         # 自动带上 [abc123]
    with log_context(action="login"):
        log.info("执行登录")     # 上下文可嵌套叠加
# 退出后上下文自动恢复
```

### 7.2 自动绑定

`app.py` 的 `do_GET` / `do_POST` 在每个请求开始时自动生成 12 位 hex 的 request_id 并绑定：

```python
def do_POST(self):
    rid = self._start_request_context()  # 绑定 request_id
    try:
        # ... 业务逻辑 ...
        log.info("POST %s", path)        # 日志自动带 [rid]
    finally:
        self._end_request_context()      # 清理上下文
```

效果：同一请求的所有日志（app → db → auth → crypto）都带相同的 request_id，便于链路追踪。

## 八、在业务代码中使用

### 8.1 标准用法

```python
from logger import get_logger
log = get_logger("my_module")

log.debug("调试信息 | user=%s", username)
log.info("操作成功 | id=%d", resource_id)
log.warning("配置缺失 | field=%s", field_name)
log.error("操作失败 | reason=%s", reason)
log.exception("捕获异常")  # 自动带堆栈
```

### 8.2 业务代码中已有的 print 替换

```python
# 旧:
print(" 已加载 %d 门课程" % len(courses))
print(" [警告] 数据加载失败: %s" % e)

# 新:
log.info("数据加载 | %d 门课程", len(courses))
log.warning("数据加载失败 | %s", e)
```

### 8.3 集成状态

| 模块 | 集成前 | 集成后 |
|------|--------|--------|
| `app.py` | `print` + `sys.stderr.write` | `get_logger("app")` + 请求级 request_id 绑定 |
| `db.py` | `print` | `get_logger("db")` 在连接/CRUD 关键点记录 |
| `auth.py` | `print` | `get_logger("auth")` 记录登录/注册成功与失败 |
| `crypto.py` | `print(file=sys.stderr)` | `get_logger("crypto")` 启动告警 |
| `llm_client.py` | 独立 logger | 保持原状（已用 logging，自动接入统一系统） |

## 九、运行时管理

### 9.1 热更新日志级别

```python
from logger import get_manager
get_manager().update_level("DEBUG")  # 不重启服务调整级别
```

### 9.2 查询日志系统状态

```python
from logger import get_manager
status = get_manager().get_status()
# {
#   "env": "prod", "level": "INFO", "log_dir": "...",
#   "console_enabled": true, "file_enabled": true,
#   "file_rolling": "time", "file_backup_count": 30,
#   "error_file_enabled": true, "handlers_count": 3
# }
```

### 9.3 程序退出时清理

`app.py` 在 `KeyboardInterrupt` 时调用 `shutdown_logging()`，确保缓冲日志刷盘：

```python
try:
    server.serve_forever()
except KeyboardInterrupt:
    log.info("收到中断信号，正在停止服务...")
    server.shutdown()
    log.info("服务已停止")
    shutdown_logging()
```

## 十二、与 llm_client.py 的关系

`llm_client.py` 在日志系统引入前已自带 `logging.getLogger("llm_client")`，会自动接入统一日志系统：
- 共享 root logger 的 handler（控制台 + 文件）
- 受全局 level 控制
- 不需要修改 `llm_client.py` 任何代码

效果：所有 LLM 调用日志（请求参数、响应、重试、熔断）都会自动持久化到 `logs/app.log`。
